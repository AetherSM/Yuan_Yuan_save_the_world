# 项目技术栈与面试问答（校园生活服务平台）

## 1. 项目一句话介绍

这是一个基于 `Spring Boot + MyBatis + MySQL + Redis + Vue3 + Vite` 的校园生活服务平台，覆盖用户认证、跑腿、商城、优惠券、地址管理、投诉、退款、后台管理等模块。

---

## 2. 技术栈总览

## 后端

- `Java 17`
- `Spring Boot 3.5.x`（Web、AOP、Mail、Test）
- `MyBatis 3.0.5`（XML Mapper）
- `MySQL`（核心业务数据存储）
- `Redis`（验证码短期缓存）
- `JWT (jjwt 0.11.5)`（登录态认证）
- `Springdoc OpenAPI`（接口文档）
- `Lombok`

## 前端

- `Vue 3`（Composition API）
- `Vite 7`
- `Vue Router 4`
- `Axios`
- `Element Plus`
- `ECharts`
- `xlsx`

## 工程与架构风格

- 前后端分离架构
- RESTful API
- 分层结构：`Controller -> Service -> Mapper -> MySQL`
- 统一返回结构 `Result<T>`
- 全局异常处理 `@RestControllerAdvice`
- JWT 拦截器 + 路由守卫的双端鉴权

---

## 3. 你可以重点讲的项目亮点

1. **鉴权链路完整**：前端路由守卫 + Axios 请求头注入 token + 后端拦截器校验 token。
2. **RBAC 角色控制**：普通用户、跑腿员、商家、管理员多角色访问限制。
3. **验证码闭环**：邮箱验证码写入 Redis（5 分钟过期），注册时校验并删除。
4. **统一错误治理**：后端全局异常处理把鉴权异常、业务异常统一成可读响应。
5. **业务模块较完整**：跑腿 + 商城 + 优惠券 + 后台管理，具备毕业设计答辩深度。

## 4. 面试官高频提问 + 参考回答

## A. Spring Boot / 分层设计

### Q1：你的后端分层是怎么设计的？

**答：**  
我按经典三层结构设计：  

- `Controller` 负责参数接收、返回结果；  
- `Service` 写业务逻辑和事务控制；  
- `Mapper` 通过 MyBatis 执行 SQL。  
这样做的好处是职责清晰、方便单元测试，也利于后续模块扩展。

### Q2：为什么选择 Spring Boot？

**答：**  
因为它开箱即用，自动配置多，能快速搭建 REST API，并且生态完整（AOP、邮件、测试、Redis、OpenAPI 都能无缝接入），非常适合这种中型业务项目。

---

## B. MyBatis / 数据访问

### Q3：你为什么选 MyBatis，而不是 JPA？

**答：**  
这个项目业务 SQL 相对复杂，尤其是筛选、统计和动态查询场景多。MyBatis 对 SQL 可控性更高，性能调优也更直观，所以我优先选了 MyBatis XML。

### Q4：你项目里怎么做动态 SQL？

**答：**  
在 Mapper XML 里使用了 `if / where / set / foreach` 等标签实现条件拼接，比如按状态、角色、时间区间做组合筛选，避免写大量重复 SQL。

---

## C. JWT 认证与权限

### Q5：JWT 在你项目里的完整流程是什么？

**答：**  

1. 用户登录后后端签发 JWT；  
2. 前端把 token 存本地并通过 Axios 请求拦截器放到请求头；  
3. 后端 `JwtInterceptor` 校验 token，有效则把 `userId` 写入请求上下文；  
4. 对 `/admin/`** 路径会额外校验管理员角色；  
5. token 无效时统一返回 401。

### Q6：你是怎么做“管理员接口保护”的？

**答：**  
后端拦截器对 `/admin` 前缀做二次权限判断：除了 token 合法，还会查用户类型是否为管理员。前端路由守卫也会先挡一层，做到前后端双保险。

### Q7：JWT 有什么缺点？你怎么规避？

**答：**  
缺点是服务端默认无状态，主动失效某个 token 不方便。我的方案是短有效期 + 前端 401 自动清理登录态并跳转登录页。进一步可升级为 Redis 黑名单机制。

---

## D. Redis / 验证码场景

### Q8：Redis 在这个项目里做了什么？

**答：**  
主要用于邮箱验证码：发送验证码时写入 `user:verify_code:{email}`，TTL 5 分钟；注册时先比对，再删除，保证验证码一次性使用和时效性。

### Q9：为什么验证码要放 Redis，不放 MySQL？

**答：**  
验证码是短时临时数据，读写频繁、无需强持久化。Redis 延迟低且天然支持过期时间，特别适合验证码这类场景。

---

## E. 异常处理 / 接口规范

### Q10：你怎么做统一异常处理？

**答：**  
使用 `@RestControllerAdvice + @ExceptionHandler`，对参数异常、权限异常、JWT 异常、系统异常统一封装成 `Result` 返回，减少 Controller 里的重复 try-catch。

### Q11：统一返回体有什么价值？

**答：**  
前后端约定固定字段（如 code/message/data），前端处理成功失败逻辑更简单，接口联调效率更高。

---

## F. 前端 Vue3 / 路由 / 网络层

### Q12：Vue3 你主要用了什么能力？

**答：**  
主要用 Composition API（`ref/onMounted/watch`）管理页面状态，按业务页面划分组件，配合 Element Plus 快速搭建后台和业务表单。

### Q13：你怎么做前端权限控制？

**答：**  
在 `router.beforeEach` 里按 `requiresAuth/requiresAdmin` 和 `userType` 做路由拦截；未登录跳登录页，不同角色限制访问不同模块。

### Q14：Axios 拦截器你做了什么？

**答：**  
请求拦截器统一加 token；响应拦截器处理 401，自动清除本地登录信息并重定向登录页，避免每个页面重复写鉴权逻辑。

---

## G. 事务 / 一致性 / 安全

### Q15：你在哪些地方用了事务？为什么？

**答：**  
典型如注册流程用了 `@Transactional`，因为里面有验证码校验、用户创建等多个步骤，任何一步失败都要回滚，保证数据一致性。

### Q16：密码安全怎么处理的？

**答：**  
当前项目使用了 MD5 做密码摘要，这是基础做法。面试时我会主动说明：线上应升级为 `BCrypt/Argon2`（含盐、可调成本），提高抗暴力破解能力。

### Q17：你项目有哪些可优化点？

**答：**  

1. 密码从 MD5 升级到 BCrypt；  
2. token 增加刷新机制和黑名单；  
3. 对热点接口加缓存和限流；  
4. 完善自动化测试与 CI；  
5. 敏感配置改用环境变量和密钥管理，不放在仓库明文。

---

## 5. 业务模块面试题（结合你这个项目）

### Q18：这个项目最核心的业务闭环是什么？

**答：**  
我会重点讲“用户登录 -> 发布跑腿/下单 -> 订单流转 -> 评价/投诉/退款 -> 管理端处理”的完整链路，展示我对真实业务流程的理解，而不仅是 CRUD。

### Q19：你如何处理多角色并存（用户/商家/跑腿员/管理员）？

**答：**  
后端根据用户类型做接口访问控制，前端根据 userType 进行菜单和路由限制。这样既保证安全，也能提升体验，避免用户看到与自己无关的功能。

### Q20：如果并发量上来，你准备怎么扩展？

**答：**  
先做读写分离和索引优化；热点数据走 Redis；无状态服务可水平扩容；再逐步引入消息队列解耦耗时任务（如通知、统计）。
